# 20.4 — C++ Code Generation Module

**Goal:** Implement a SymPy-to-C++ code generator that produces stencil `.cpp` and `.t.cpp` files matching the existing hand-written patterns in `src/stencils/`. The generated code must be numerically identical to the existing implementations when evaluated at the same alpha (and psi) values.

**Depends on:**
- 20.2 (interior stencil derivation -- provides `InteriorCoefficients`)
- 20.3 (uniform boundary stencil derivation -- provides boundary coefficient expressions as functions of alpha)

**Read first:**
- `src/stencils/E4u_1.cpp` -- simplest uniform stencil, canonical code structure
- `src/stencils/polyE2_1.cpp` -- cut-cell with psi-dependent CSE temporaries
- `src/stencils/E8u_1.cpp` (first 120 lines) -- largest uniform, struct pattern and interior
- `src/stencils/stencil.hpp` -- `Stencil` concept, `info` struct, factory signatures
- `src/stencils/E4u_1.t.cpp` -- Catch2 test file pattern
- `plans/stencil-derivation-math-reference.md` (Section 7: Output Format)

**Test commands:**
```bash
cd scripts/stencil_gen && uv run pytest tests/test_printer.py -v
cd scripts/stencil_gen && uv run pytest tests/test_codegen.py -v
cd scripts/stencil_gen && uv run pytest tests/test_codegen_e4u.py -v
```

### Dependency interfaces (from 20.2 and 20.3)

20.4g imports from `stencil_gen.interior` and `stencil_gen.boundary` (built by
20.2 and 20.3 respectively). The expected APIs are:

```python
# From 20.2 (interior.py):
@dataclass
class InteriorCoefficients:
    gamma: list[Rational]   # length 2p+1, exact Rational coefficients (no h)
    # (other fields not used by codegen)

def derive_interior(s: int, p: int, nu: int) -> InteriorCoefficients: ...

# From 20.3 (boundary.py):
@dataclass
class BoundaryResult:
    floating_coeffs: list[Expr]   # length R*T, symbolic in alpha_{0..n-1} (and psi for cut-cell)
    dirichlet_coeffs: list[Expr]  # length R*T, symbolic (row 0 present but skipped for Dirichlet output)
    # (other fields not used by codegen)

def derive_boundary(s: int, p: int, r: int, nu: int) -> BoundaryResult: ...
```

**Key convention for `dirichlet_coeffs`:** The derivation module returns all R*T
coefficients (R rows × T columns, row-major). For Dirichlet boundary conditions,
row 0 is the Dirichlet constraint row and is NOT emitted in the C++ code. The
code generator must slice `dirichlet_coeffs[T:]` to get the (R-1)*T coefficients
for `nbs_dirichlet`. See 20.4d Step 1 and 20.4e Section 4i.

## 20.4a -- Custom C++ printer (`printer.py`)

**File:** `scripts/stencil_gen/stencil_gen/printer.py`
**Test file:** `scripts/stencil_gen/tests/test_printer.py`
**Test:** `cd scripts/stencil_gen && uv run pytest tests/test_printer.py -v`

Implement `StencilCodePrinter(C99CodePrinter)` that converts SymPy expressions into
C++ source fragments matching the style in the existing stencil files.

### Constructor

```python
class StencilCodePrinter(C99CodePrinter):
    def __init__(self, symbol_map: dict[Symbol, str] | None = None):
        super().__init__(settings={"precision": 17})
        self._symbol_map = symbol_map or {}
```

The `symbol_map` maps SymPy `Symbol` objects to C++ variable strings. Built by
`build_symbol_map()` (below).

### Printing rules with before/after examples

Each rule corresponds to a `_print_*` method override on the printer class.

**1. `Pow(x, -1)` -- reciprocal:**

| SymPy | Default C99 output | Our output |
|-------|-------------------|------------|
| `Pow(1 + psi, -1)` | `1.0/(1 + psi)` or `pow(1 + psi, -1)` | `1.0 / (1 + psi)` |

Implementation: in `_print_Pow`, check `expr.exp == -1`; emit `1.0 / (%s)` where
`%s` is the printed base. This matches the `real t6 = 1.0 / (t5);` pattern in
`polyE2_1.cpp` line 177.

**2. `Pow(x, 2)` -- square:**

| SymPy | Default C99 output | Our output |
|-------|-------------------|------------|
| `Pow(psi, 2)` | `pow(psi, 2)` | `psi * psi` |

Implementation: check `expr.exp == 2`; emit `%s * %s` with the printed base.
Parenthesize the base if it is an Add or Mul (check with `expr.base.is_Add` or
`expr.base.is_Mul`).

**3. `Pow(x, 3)` -- cube:**

| SymPy | Default C99 output | Our output |
|-------|-------------------|------------|
| `Pow(psi, 3)` | `pow(psi, 3)` | `psi * psi * psi` |

Implementation: check `expr.exp == 3`; emit `%s * %s * %s`. Same parenthesization
rule as square.

**4. `Pow(x, n)` for integer n >= 4 -- use std::pow:**

| SymPy | Default C99 output | Our output |
|-------|-------------------|------------|
| `Pow(psi, 5)` | `pow(psi, 5)` | `std::pow(psi, 5)` |

Implementation: for `expr.exp` integer >= 4, emit `std::pow(%s, %d)`. The parent
`C99CodePrinter` emits `pow(...)` (from `<cmath>`) -- override to use `std::pow`.

**5. `Pow(x, n)` for negative integer n <= -2 -- reciprocal of positive power:**

| SymPy | Default C99 output | Our output |
|-------|-------------------|------------|
| `Pow(1 + psi, -2)` | `pow(1 + psi, -2)` | `1.0 / ((1 + psi) * (1 + psi))` |

Implementation: check `expr.exp` is integer < -1; emit `1.0 / (%s)` where `%s` is
the printed `Pow(base, -exp)` (recursion handles the positive exponent case).

**6. `Rational(p, q)` where q != 1 -- float literal division:**

| SymPy | Default C99 output | Our output |
|-------|-------------------|------------|
| `Rational(1, 12)` | `1.0/12.0` or `1.0L/12.0L` | `1.0 / 12` |
| `Rational(2, 3)` | `2.0/3.0` | `2.0 / 3` |
| `Rational(-5, 6)` | `-5.0/6.0` | `-5.0 / 6` |

Implementation: in `_print_Rational`, emit `%s.0 / %d` where the numerator gets
`.0` suffix (to force `double` promotion) and the denominator is a plain integer.
This matches the existing style: `1 / (12 * h)` in E4u_1 line 51 uses integer
division context (h is `real`), but standalone rationals like `(6 * alpha[0] - 11) / 6`
use the mixed int/real pattern. For negative rationals, put the sign on the numerator:
`-5.0 / 6`.

**7. `Rational(n, 1)` -- integer literal:**

| SymPy | Default C99 output | Our output |
|-------|-------------------|------------|
| `Rational(3, 1)` or `Integer(3)` | `3` | `3` |
| `Rational(-1, 1)` or `Integer(-1)` | `-1` | `-1` |

Implementation: emit `str(p)` with no `.0` suffix. Integers used as multipliers
should be plain integers (matching `3 - 4 * alpha[0]` in E4u_1 line 82).

**8. `Symbol` -- via symbol_map:**

| SymPy | symbol_map entry | Our output |
|-------|-----------------|------------|
| `Symbol('alpha_0')` | `{alpha_0: 'alpha[0]'}` | `alpha[0]` |
| `Symbol('alpha_1')` | `{alpha_1: 'alpha[1]'}` | `alpha[1]` |
| `Symbol('fa_3')` | `{fa_3: 'fa[3]'}` | `fa[3]` |
| `Symbol('psi')` | `{psi: 'psi'}` | `psi` |
| `Symbol('h')` | `{h: 'h'}` | `h` |
| `Symbol('t5')` (CSE temp) | not in map | `t5` |

Implementation: in `_print_Symbol`, look up `expr` in `self._symbol_map`; if found
return the mapped string, otherwise return `expr.name`. CSE temporaries (`t5`, `t6`,
...) are not in the map and print as their name.

### `build_symbol_map` helper

```python
def build_symbol_map(
    param_arrays: dict[str, int],  # e.g. {"alpha": 2} or {"fa": 6, "da": 3, "ia": 4}
    has_psi: bool = False,
) -> dict[Symbol, str]:
```

Algorithm:
1. For each `(name, count)` in `param_arrays`: create `Symbol(f'{name}_{i}')` -> `f'{name}[{i}]'` for `i` in `range(count)`.
2. If `has_psi`: add `Symbol('psi')` -> `'psi'`.
3. Always add `Symbol('h')` -> `'h'`.
4. Return the dict.

Example: `build_symbol_map({"alpha": 2})` returns:

```python
{Symbol('alpha_0'): 'alpha[0]', Symbol('alpha_1'): 'alpha[1]', Symbol('h'): 'h'}
```

## 20.4b -- CSE wrapper (`codegen.py`)

**File:** `scripts/stencil_gen/stencil_gen/codegen.py`
**Test file:** `scripts/stencil_gen/tests/test_codegen.py`
**Test:** `cd scripts/stencil_gen && uv run pytest tests/test_codegen.py::test_cse -v`

### `apply_cse` function

```python
def apply_cse(
    coeffs: list[Expr],
    prefix: str = "t",
    start: int = 5,
) -> tuple[list[tuple[Symbol, Expr]], list[Expr]]:
```

Wraps `sympy.cse()` with project conventions:

```python
from sympy import cse, numbered_symbols
replacements, reduced = cse(
    coeffs,
    symbols=numbered_symbols(prefix, start=start),
)
return replacements, reduced
```

The `start=5` default matches the existing naming convention in `polyE2_1.cpp` where
CSE temporaries begin at `t5` (lines 175-186). This avoids clashing with the first
few temporaries that might be reserved for other purposes.

### When to use CSE

Decision matrix:

| Stencil type | Psi-dependent? | Use CSE? | Rationale |
|-------------|---------------|----------|-----------|
| Uniform (E4u, E6u, E8u) | No | **No** | Existing code uses inline expressions with no temporaries. Alpha terms are simple linear combinations. CSE would add unnecessary temporaries. |
| Cut-cell (polyE2, E2) | Yes | **Yes** | Psi appears in denominators creating `1/(1+psi)`, `1/(2+psi)` etc. that repeat across rows. CSE hoists these into `real t5 = 1 + psi; real t6 = 1.0 / (t5);` matching existing patterns. |
| Fallback | Either | Yes if `max(count_ops(c) for c in coeffs) > threshold` | Safety net: if any single coefficient has more than 20 operations, apply CSE even for uniform stencils. In practice this never triggers for current schemes. |

The threshold of 20 ops is a conservative heuristic. The most complex uniform
coefficient in E8u_1 has ~15 ops (line 66: seven alpha terms plus constants), well
below the threshold.

### Tests

```python
def test_cse_psi_dependent():
    """CSE hoists common psi subexpressions."""
    psi = Symbol('psi')
    a = Symbol('alpha_0')
    coeffs = [a / (1 + psi), (1 - a) / (1 + psi), a / (2 + psi)]
    repls, reduced = apply_cse(coeffs, prefix="t", start=5)
    # (1 + psi) should be hoisted as a common subexpression
    assert len(repls) >= 1
    # Verify algebraic equivalence after substitution
    for orig, red in zip(coeffs, reduced):
        restored = red.xreplace(dict(repls))
        assert (orig - restored).cancel() == 0

def test_cse_uniform_skipped():
    """Uniform stencil: CSE is not called (decision is caller's)."""
    # This test validates the decision logic, not apply_cse itself
    a0, a1 = symbols('alpha_0 alpha_1')
    coeffs = [(6*a0 - 11) / 6, 3 - 4*a0, a1]
    # For uniform, caller should NOT call apply_cse
    # Verify the expressions are simple enough
    assert all(c.count_ops() <= 20 for c in coeffs)
```

### `generate_interior_method` function

```python
def generate_interior_method(
    coeffs: list[Rational],  # length 2p+1, exact Rationals from interior derivation
    nu: int,                 # derivative order (1 or 2)
    is_uniform: bool,        # True for uniform stencils (bake h into expressions)
) -> str:
```

Returns the **body only** (lines between `{` and `}`) of the `interior(real h,
std::span<real> c) const` method. The caller (`generate_stencil_cpp` in 20.4e)
wraps it in the method signature. Each line in the returned string is indented
with 8 spaces (matching the existing 2-level indent inside struct + method).

**Algorithm:**

1. Let `P = (len(coeffs) - 1) // 2`.
2. Determine symmetry type:
   - nu=1 (first derivative): coefficients are antisymmetric -- `c[2P-k] = -c[k]` and `c[P] = 0`
   - nu=2 (second derivative): coefficients are symmetric -- `c[2P-k] = c[k]`
3. Choose code pattern based on `is_uniform`:

**Uniform stencils (baked-in h division):**

Match the E4u_1 and E8u_1 pattern where `/h` is part of each expression:

```cpp
// E4u_1 interior (nu=1, P=2):
c[0] = 1 / (12 * h);        // Rational(1,12) / h
c[1] = -2 / (3 * h);        // Rational(-2,3) / h
c[2] = 0;                   // center is always 0 for nu=1
c[3] = -c[1];               // antisymmetry shorthand
c[4] = -c[0];               // antisymmetry shorthand

return c.subspan(0, 2 * P + 1);
```

For the first P+1 coefficients, emit `c[i] = <rational> / h;` (or `c[i] = <rational> / (h * h);` for nu=2). The rational is printed as `p / (q * h)` if q != 1, or `1 / h` if p/q == 1.

Then emit the symmetric/antisymmetric tail using shorthand:
- nu=1: `c[2P] = -c[0]; c[2P-1] = -c[1]; ...` (and `c[P] = 0;`)
- nu=2: `c[2P] = c[0]; c[2P-1] = c[1]; ...`

**Cut-cell stencils (loop-based h division):**

Match the polyE2_1 pattern:

```cpp
// polyE2_1 interior (nu=1, P=1):
c = c.subspan(0, 2 * 1 + 1);
c[0] = -0.5;
c[1] = 0;
c[2] = 0.5;
for (auto&& v : c) v /= h;
return c;
```

For cut-cell, emit all 2P+1 coefficients explicitly (no shorthand), then
the h-division loop. For nu=1: `for (auto&& v : c) v /= h;`. For nu=2:
`for (auto&& v : c) v /= (h * h);`. (Currently all cut-cell stencils in
scope are nu=1, so the nu=2 path is included for completeness.)

**Detection of shorthand eligibility:**

The shorthand `c[2P-k] = -c[k]` is used when:
- The stencil is uniform (is_uniform=True)
- nu=1 (antisymmetric): verify `coeffs[2P-k] == -coeffs[k]` for all k
- nu=2 (symmetric): verify `coeffs[2P-k] == coeffs[k]` for all k

If the symmetry check fails (should not happen for correctly derived stencils),
fall back to explicit assignment for all 2P+1 coefficients.

### `format_rational_h_division` helper

```python
def format_rational_h_division(r: Rational, nu: int) -> str:
    """Format a rational coefficient with baked-in h-division for interior methods.

    Returns a C++ expression string. Not using StencilCodePrinter because these
    are exact Rational constants, not general symbolic expressions.
    """
```

**Algorithm:**
1. If `r == 0`, return `"0"`.
2. Let `p, q = int(r.p), int(r.q)` (sign on `p`, `q` always positive).
3. Build the h-divisor string: `"h"` for nu=1, `"(h * h)"` for nu=2.
4. If `q == 1`: return `f"{p} / {h_div}"`.
5. If `q != 1`: return `f"{p} / ({q} * {h_div})"`.

**nu=1 format table (from E4u_1.cpp, E8u_1.cpp):**

| Rational | C++ expression |
|---------|---------------|
| `Rational(1, 12)` | `1 / (12 * h)` |
| `Rational(-2, 3)` | `-2 / (3 * h)` |
| `Rational(1, 5)` | `1 / (5 * h)` |
| `Rational(-4, 5)` | `-4 / (5 * h)` |
| `Rational(1, 1)` | `1 / h` |
| `Rational(0, 1)` | `0` |

**nu=2 format table (from E2_2.cpp, E4_2.cpp):**

| Rational | C++ expression |
|---------|---------------|
| `Rational(1, 1)` | `1 / (h * h)` |
| `Rational(-2, 1)` | `-2 / (h * h)` |
| `Rational(-1, 12)` | `-1 / (12 * (h * h))` |
| `Rational(4, 3)` | `4 / (3 * (h * h))` |
| `Rational(-5, 2)` | `-5 / (2 * (h * h))` |
| `Rational(0, 1)` | `0` |

## 20.4d -- Boundary method generator (`codegen.py`)

**File:** `scripts/stencil_gen/stencil_gen/codegen.py`
**Test:** `cd scripts/stencil_gen && uv run pytest tests/test_codegen.py::test_nbs -v`

### `generate_nbs_method` function

```python
def generate_nbs_method(
    method_name: str,          # "nbs_floating" or "nbs_dirichlet"
    coeffs: list[Expr],        # r*t (floating) or (r-1)*t (Dirichlet) expressions
    r: int,
    t: int,
    printer: StencilCodePrinter,
    psi_dependent: bool,       # True for cut-cell stencils
    nu: int = 1,               # derivative order (affects right-boundary logic)
) -> str:
```

Returns the **complete method** including signature and braces. The method
is indented with 4 spaces (one level inside the struct). Body lines use 8 spaces.

### Algorithm

**Step 1: Determine subspan size**

```python
if method_name == "nbs_floating":
    n_coeffs = r * t
    span_expr = "R * T"
else:  # nbs_dirichlet
    n_coeffs = (r - 1) * t
    span_expr = "(R - 1) * T"
```

**Step 2: Optionally apply CSE**

```python
if psi_dependent:
    replacements, reduced = apply_cse(coeffs)
else:
    replacements = []
    reduced = coeffs
```

**Step 3: Emit method signature**

For uniform stencils (psi not used):
```cpp
    std::span<const real> nbs_floating(real h, real, std::span<real> c, bool right) const
    {
```
Note the unnamed `real` parameter -- psi is accepted but ignored. This matches
E4u_1 line 77 and E8u_1 line 81.

Formatting detail: the hand-written code is inconsistent about blank lines after
the opening `{` (E4u_1 `nbs_floating` has one, E8u_1 does not). The generator
should emit NO blank line after `{` for consistency.

For cut-cell stencils (psi used):
```cpp
std::span<const real>
nbs_floating(real h, real psi, std::span<real> c, bool right) const
{
    c = c.subspan(0, R * T);
```
Note the named `psi` parameter and the `c.subspan()` call. This matches
polyE2_1 lines 170-173.

**Step 4: Emit CSE temporaries (if any)**

For each `(sym, expr)` in `replacements`:
```cpp
        real t5 = 1 + psi;
        real t6 = 1.0 / (t5);
        real t7 = fa[1];
        ...
```

Use `printer.doprint(expr)` for the right-hand side. This matches polyE2_1
lines 175-186.

Note: insert a blank line between the temporaries and the coefficient assignments.

**Step 5: Emit coefficient assignments**

For each index `i`, emit:
```cpp
        c[0] = <expr>;
        c[1] = <expr>;
        ...
```

Use `printer.doprint(reduced[i])` for each expression.

For uniform stencils, the expressions contain alpha symbols directly (no CSE temps):
```cpp
        c[0] = (6 * alpha[0] - 11) / 6;
        c[1] = 3 - 4 * alpha[0];
```
This matches E4u_1 lines 80-94.

For cut-cell stencils, the expressions reference CSE temporaries:
```cpp
        c[0] = 0.5 * t6 * t8;
        c[1] = (-1 + t10) * 0.5;
```
This matches polyE2_1 lines 187-198.

**Step 6: Emit h-division loop**

Parameterized by derivative order (`nu`):

For nu=1 (first derivative):
```cpp
        for (auto&& v : c) v /= h;
```
(Matches E4u_1 line 96, polyE2_1 line 200.)

For nu=2 (second derivative):
```cpp
        for (auto&& v : c) v /= (h * h);
```
(Matches E2_2 line 131, E4_2 line 232. Note the parentheses around `h * h`.)

**Step 7: Emit right-boundary logic**

For first-derivative stencils (nu=1):
```cpp
        if (right) {
            for (auto&& v : c) v *= -1;
            std::ranges::reverse(c);
        }
```
(Negate then reverse -- matches E4u_1 lines 97-100, polyE2_1 lines 201-204.)

For second-derivative stencils (nu=2):
```cpp
        if (right) {
            std::ranges::reverse(c);
        }
```
(Reverse only, no negation.)

**Step 8: Return statement**

```cpp
        return c;
```

### Floating vs Dirichlet sizing

The `nbs()` dispatcher method subspans differently for each BC type:

- Floating: `c.subspan(0, R * T)` -- all r rows contribute
- Dirichlet: `c.subspan(0, (R - 1) * T)` -- first row is dropped (the Dirichlet
  boundary value replaces it)

This is visible in E4u_1 lines 69-71: floating uses `R * T`, Dirichlet uses `(R - 1) * T`.

The boundary derivation module (20.3) produces r rows of coefficients. For Dirichlet,
the code generator should skip row 0 and emit rows 1 through r-1 only. Concretely,
`generate_stencil_cpp` (20.4e) slices `spec.dirichlet_coeffs[T:]` before passing to
`generate_nbs_method`, so the `coeffs` parameter in `generate_nbs_method` always has
exactly the number of entries to emit ((R-1)*T for Dirichlet).

### Code skeleton that gets filled in

Complete method template for `nbs_floating` on a uniform stencil (E4u_1 pattern):

```cpp
    std::span<const real> nbs_floating(real h, real, std::span<real> c, bool right) const
    {
{%- if not is_uniform %}
        c = c.subspan(0, R * T);
{% endif %}
{%- for sym, expr in cse_replacements %}
        real {{ sym }} = {{ expr }};
{%- endfor %}
{%- if cse_replacements %}

{%- endif %}
{%- for i, expr in enumerate(coefficients) %}
        c[{{ i }}] = {{ expr }};
{%- endfor %}

{%- if nu == 1 %}
        for (auto&& v : c) v /= h;
        if (right) {
            for (auto&& v : c) v *= -1;
            std::ranges::reverse(c);
        }
{%- elif nu == 2 %}
        for (auto&& v : c) v /= (h * h);
        if (right) {
            std::ranges::reverse(c);
        }
{%- endif %}

        return c;
    }
```

(The template notation is for illustration only; the actual implementation uses
Python string formatting, not Jinja.)

### Algorithm for `generate_stencil_cpp`

The function builds the output string by concatenating sections in order. Each
section is described below with its decision logic.

**Step 1: Build printer and symbol map**

```python
smap = build_symbol_map(spec.param_arrays, has_psi=not spec.is_uniform)
printer = StencilCodePrinter(symbol_map=smap)
```

**Step 2: Emit includes (Section 1)**

Always the same three includes, separated by blank lines:
```cpp
#include "stencil.hpp"\n\n#include <algorithm>\n\n#include <cmath>\n
```

**Step 3: Emit optional comment (Section 2)**

For uniform stencils only: extract the family name by stripping the trailing
`_1` or `_2` from `spec.name` (e.g., `"E4u_1"` → `"E4u"`).
```python
if spec.is_uniform:
    family = spec.name.rsplit("_", 1)[0]
    emit(f"/// {family} - uniform mesh {family} stencil\n\n")
```

**Step 4: Emit namespace + struct open (Section 3, 4)**

```cpp
namespace ccs::stencils\n{\nstruct {name} {\n
```

**Step 5: Emit constants (Section 4a)**

Four `static constexpr int` lines for P, R, T, X.

**Step 6: Emit member arrays (Section 4b)**

For each `(array_name, count)` in `spec.param_arrays` (in insertion order):
```cpp
    std::array<real, {count}> {array_name};
```

**Step 7: Emit constructors (Section 4c)**

- Always emit `{name}() = default;`
- For uniform stencils (single array, e.g. `{"alpha": 2}`):
  ```cpp
  {name}(std::span<const real> a)
  {
      copy_zero_padded(a, {array_name});
  }
  ```
- For cut-cell stencils (multiple arrays, e.g. `{"fa": 6, "da": 3, "ia": 4}`):
  one parameter per array with `_` suffix, one `copy_zero_padded` call per array.

**Step 8: Emit query methods (Section 4d)**

- `query_max()` returns `{P, R, T, X}`.
- `query(bcs::type b)` switch: Dirichlet → `{P, R-1, T, 0}`, Floating → `{P, R, T, 0}`,
  Neumann → `{}`, default → `{}`.
- `query_interp()`: if `spec.has_interp` → `{spec.interp_P, spec.interp_T}`, else `{}`.

**Step 9: Emit interpolation stubs (Section 4e)**

For 20.4, always emit the stub (no-op) versions:
- `interp_interior`: unnamed `real` param, returns `c`.
- `interp_wall`: unnamed `int, real, real` params, returns `c`.

Full interp generation for cut-cell stencils (polyE2_1-style `interp_wall` with
CSE temporaries and switch/case) is deferred to a future phase.

**Step 10: Emit interior method (Section 4f)**

Call `generate_interior_method(spec.interior_coeffs, spec.derivative_order,
spec.is_uniform)` and wrap in the method signature:
```cpp
    std::span<const real> interior(real h, std::span<real> c) const
    {
        {body}
    }
```

**Step 11: Emit nbs() dispatcher (Section 4g)**

Decision: if `spec.is_uniform`, subspan in the dispatcher (passing pre-sliced
span to helpers). If cut-cell, pass `c` directly (helpers do their own subspan).

For uniform:
```cpp
        case bcs::Floating:
            return nbs_floating(h, psi, c.subspan(0, R * T), right);
        case bcs::Dirichlet:
            return nbs_dirichlet(h, psi, c.subspan(0, (R - 1) * T), right);
```
For cut-cell:
```cpp
        case bcs::Floating:
            return nbs_floating(h, psi, c, right);
        case bcs::Dirichlet:
            return nbs_dirichlet(h, psi, c, right);
```

The `x`/unnamed parameter: use `std::span<real> x` for uniform,
`std::span<real>` (unnamed) for cut-cell.

**Step 12: Emit nbs_floating and nbs_dirichlet (Sections 4h, 4i)**

For floating:
```python
floating_body = generate_nbs_method("nbs_floating", spec.floating_coeffs,
    spec.R, spec.T, printer, psi_dependent=not spec.is_uniform, nu=spec.derivative_order)
```

For Dirichlet — slice to skip row 0:
```python
dirichlet_coeffs = spec.dirichlet_coeffs[spec.T:]  # skip first T entries (row 0)
dirichlet_body = generate_nbs_method("nbs_dirichlet", dirichlet_coeffs,
    spec.R, spec.T, printer, psi_dependent=not spec.is_uniform, nu=spec.derivative_order)
```

**Step 13: Emit nbs_neumann stub (Section 4j)**

Decision rule:
- Uniform first-derivative (is_uniform and X == 0): void-returning stub:
  ```cpp
  void nbs_neumann(real, real, std::span<real>, std::span<real>, bool) const {}
  ```
- Cut-cell / Neumann-capable (not is_uniform or X > 0): span-returning stub:
  ```cpp
  std::span<const real>
  nbs_neumann(real, real, std::span<real> c, std::span<real>, bool) const
  {
      return c;
  }
  ```

**Step 14: Emit struct close + factory function (Section 5)**

For uniform (single `param_arrays` entry):
```cpp
stencil make_{name}(std::span<const real> {array_name}) {{ return {name}{{{array_name}}}; }}
```

For cut-cell (multiple entries): one parameter per array, construct with all arrays.

**Step 15: Emit namespace close (Section 6)**

```cpp
}} // namespace ccs::stencils
```

### Key decision points in the generator

**nbs() dispatcher -- subspan location:**

For uniform stencils (E4u_1, E8u_1), the `nbs()` method subspans _before_ calling
the helper: `c.subspan(0, R * T)`. The helper receives the pre-sliced span.

For cut-cell stencils (polyE2_1), the `nbs()` method passes `c` directly, and the
helper does its own subspan: `c = c.subspan(0, R * T);`.

Decision rule: if `is_uniform`, subspan in the dispatcher. Otherwise, subspan in
the helper.

**nbs_neumann stub style:**

For uniform first-derivative stencils, use the void-returning stub:
```cpp
void nbs_neumann(real, real, std::span<real>, std::span<real>, bool) const {}
```

For cut-cell stencils, use the span-returning stub:
```cpp
std::span<const real>
nbs_neumann(real, real, std::span<real> c, std::span<real>, bool) const
{
    return c;
}
```

**Extra parameter name (x vs unnamed):**

The `nbs()` method's last parameter is `std::span<real> x` in uniform stencils
(E4u_1 line 65) and unnamed `std::span<real>` in cut-cell (polyE2_1 line 156).
The generator should use `x` for uniform and unnamed for cut-cell.

### Implementation decomposition

`generate_stencil_cpp` is ~250 lines of string assembly. Break it into these
internal helper functions within `codegen.py` to keep each piece testable:

| Helper function | Sections | ~Lines | What it emits |
|----------------|----------|--------|---------------|
| `_emit_header(spec)` | 1–3 | 15 | Includes, optional comment, namespace open |
| `_emit_struct_preamble(spec)` | 4a–4c | 30 | Constants, member arrays, constructors |
| `_emit_query_methods(spec)` | 4d–4e | 25 | `query_max`, `query`, `query_interp`, interp stubs |
| `_emit_interior_method(spec)` | 4f | 10 | Wraps `generate_interior_method()` in signature |
| `_emit_nbs_dispatcher(spec)` | 4g | 20 | The `nbs()` switch dispatcher |
| `_emit_nbs_methods(spec, printer)` | 4h–4j | var | Calls `generate_nbs_method()` for floating/Dirichlet + neumann stub |
| `_emit_factory(spec)` | 5–6 | 10 | Struct close, factory function, namespace close |

`generate_stencil_cpp` calls these in order and concatenates. The helpers are
private (`_` prefix) and share the `StencilGenSpec` parameter. Only
`generate_stencil_cpp` is the public API.

**Critical detail in `_emit_nbs_methods`:** Dirichlet coefficients must be sliced
to skip row 0 before passing to `generate_nbs_method`:
```python
dirichlet_coeffs = spec.dirichlet_coeffs[spec.T:]  # skip first T entries (row 0)
```

### Scoped block and sizing details

Each `TestCase` generates one anonymous scoped block `{ ... }` within the
`TEST_CASE`. All blocks share the Lua setup and stencil construction.

**`ex` vector:** Always `T ex{}` (empty vector). The extra parameter is only used

the `TestCase`.

**`right` parameter:** Always `false` in generated tests. Right-boundary behavior
is validated by symmetry in the C++ test suite, not by the Python generator.

### Generating numerical test data

The test data (`expected_coeffs`) is computed by evaluating the symbolic boundary
expressions at concrete alpha (and psi) values. This happens at generation time in
Python:

```python
def compute_test_values(
    coeffs: list[Expr],          # symbolic expressions
    alpha_values: dict[str, list[float]],  # concrete alpha values
    h: float,
    psi: float,
    right: bool = False,
    nu: int = 1,
) -> list[float]:
    """Evaluate symbolic coefficients at given parameter values."""
    # Build substitution map
    subs = {}
    for name, values in alpha_values.items():
        for i, v in enumerate(values):
            subs[Symbol(f'{name}_{i}')] = v
    subs[Symbol('psi')] = psi
    subs[Symbol('h')] = h

    # Evaluate
    result = [float(c.xreplace(subs)) for c in coeffs]

    # Apply h division (parameterized by derivative order)
    h_divisor = h if nu == 1 else h * h
    result = [v / h_divisor for v in result]

    # Apply right-boundary transformation if needed
    if right and nu == 1:
        result = [-v for v in reversed(result)]
    elif right and nu == 2:
        result = list(reversed(result))

    return result
```

## 20.4g -- End-to-end E4u_1 validation (`tests/test_codegen_e4u.py`)

**File:** `scripts/stencil_gen/tests/test_codegen_e4u.py`
**Test:** `cd scripts/stencil_gen && uv run pytest tests/test_codegen_e4u.py -v`

### Purpose

This is the integration test that validates the full pipeline from symbolic
derivation through C++ code generation. It ensures the generated code is
numerically equivalent to the existing hand-written `src/stencils/E4u_1.cpp`.

### Blocking dependency on 20.2 and 20.3

This test file imports `derive_interior` from `stencil_gen.interior` (phase 20.2)
and `derive_boundary` from `stencil_gen.boundary` (phase 20.3). **These modules
must be implemented before 20.4g tests can run.** Phases 20.4a–20.4f can be
implemented and tested independently using hand-constructed symbolic expressions
(no derivation imports needed).

If 20.2/20.3 are not yet available, the 20.4g test file can be written but will
fail on import. Use `pytest.importorskip("stencil_gen.interior")` as a guard if
needed during development.

### Strategy

**Do NOT compare strings.** The generated code may differ from the existing code
in superficial ways:
- CSE may order temporaries differently
- SymPy may factor expressions differently (`(6 * alpha[0] - 11) / 6` vs
  `alpha[0] - 11.0/6`)
- Whitespace and comment differences

Instead, validate **numerical equivalence**: compile and run both implementations
with the same inputs, compare outputs.

### Why numerical comparison, not string comparison

1. **CSE ordering is non-deterministic** across SymPy versions. `sympy.cse()` may
   assign `t5` to `1 + psi` in one version and `fa[0]` in another.

2. **Expression factoring varies.** SymPy's printer may produce
   `(6 * alpha[0] - 11) / 6` or `alpha[0] - 11.0 / 6` -- both algebraically
   correct but string-different.

3. **The contract is numerical correctness**, not syntactic identity. If the generated
   code produces the same coefficient values (within floating-point tolerance) for
   all tested alpha/psi inputs, it is correct.

4. **String comparison is fragile and would break on every SymPy upgrade.** Numerical
   comparison is stable and meaningful.

For cases where we want to verify code _structure_ (not values), use structural
assertions like `assert "struct E4u_1" in code` rather than exact string matching.

## Out of scope for 20.4

- **`from_lua` factory registration:** The generated `.cpp` file includes the
  factory function (`make_E4u_1`, etc.) but does NOT modify `stencil.cpp` to
  register it in `stencil::from_lua()`. Registration is a manual step or a
  future extension.
- **Full interpolation generation:** The `interp_interior` and `interp_wall`
  methods for cut-cell stencils (e.g., polyE2_1's 80-line `interp_wall` with
  CSE and switch/case) are not generated. Only stubs are emitted. The
  `StencilGenSpec.has_interp` and related fields are reserved for future use.
- **Neumann boundary methods:** Stencils with `X > 0` (e.g., E2_2, E4_2) have
  non-trivial `nbs_neumann` implementations. Only stubs are emitted in 20.4.
  Full Neumann generation is part of 20.5.

1. **20.4a** (printer) -- no dependencies, foundational
